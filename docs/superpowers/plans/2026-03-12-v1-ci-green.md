## Chunk 3: Test Release Workflow

The release workflow has this job dependency chain:

```
bump-version
  ↓              ↓
build-publish-pypi    build-publish-docker
  ↓              ↓
       create-release
```

`create-release` only runs if both publish jobs succeed. `build-publish-pypi` requires `PYPI_TOKEN` secret to upload to PyPI. If that secret is not configured, the publish job fails → `create-release` is blocked.

**Best test method:** `workflow_dispatch` (manual trigger) which skips PyPI upload via an `if:` condition, or trigger via a tag and accept that `build-publish-pypi` may be skipped/fail gracefully.

### Task 3: Trigger release via workflow_dispatch

- [ ] **Step 1: Check current version in pyproject.toml**

```bash
grep '^version' pyproject.toml
```
Expected: e.g. `version = "2.0.0"`

- [ ] **Step 2: Trigger workflow_dispatch with patch bump**

```bash
gh workflow run release.yml --field version_type=patch
```
Expected: `Created workflow dispatch event.`

- [ ] **Step 3: Monitor the run**

```bash
gh run list --workflow=release.yml --limit 3
gh run watch
```

Expected job sequence:
1. `bump-version` — bumps `pyproject.toml` from `2.0.0` → `2.0.1`, commits, pushes tag `v2.0.1`
2. `build-publish-pypi` — may succeed (if `PYPI_TOKEN` set) or fail (if not set)
3. `build-publish-docker` — builds Docker image
4. `create-release` — creates GitHub release with populated `release_notes`

- [ ] **Step 4: Verify `bump-version` job outputs**

```bash
gh run view <run-id>
```
Expected:
- `new_version` = `2.0.1`
- `release_notes` is NOT empty (populated from `steps.notes.outputs.release_notes`)

- [ ] **Step 5: If `build-publish-pypi` fails due to missing PYPI_TOKEN**

This is acceptable for smoke testing — the important job is `bump-version` (versioning logic) and `create-release` (GitHub release creation).

Check whether `create-release` is blocked:
```bash
gh run view <run-id> --json jobs --jq '.jobs[] | {name: .name, status: .status, conclusion: .conclusion}'
```
If `create-release` has `conclusion: skipped` due to failed dependency, the version bump still happened correctly.

- [ ] **Step 6: Verify GitHub release was created (if create-release ran)**

```bash
gh release list --limit 3
```
Expected: `v2.0.1` with non-empty body.
